Response strategy: four win themes

A utility replacing VxBlock and PowerMax is not buying hardware, it is buying certainty through 2033. The technical response should be organized so every evaluator scores these four themes on every page.

Win theme 1

Utility-grade availability, proven topology

RAC for node-level resilience, Data Guard across the three data centers for site-level resilience. Recovery targets stated as measurable SLOs (RPO, RTO per tier) instead of adjectives. Evaluators can score numbers, not promises.

Win theme 2

Compliance as evidence, not assurance

NERC CIP sections answered with a control-by-control mapping (TDE, key management, PAM, RBAC, hardening baselines, SIEM feeds) and the audit artifacts each control produces. This is what separates a utility bid from a generic IT bid.

Win theme 3

Migration as a factory, not an event

Pilot first, then repeatable waves with standard runbooks, rollback gates, and validation criteria. Wave planning by dependency and criticality. The proposal shows the machine, so the effort estimate becomes believable.

Win theme 4

Lifecycle economics to 2033

Platform choice argued on 8-year lifecycle: support horizons, capacity headroom, patching model, Oracle 26ai upgrade path, and exit options. Sizing includes stated growth buffer and DR capacity so the number survives scrutiny.

Target architecture: three sites, one recovery story

Reference topology for Oracle 19c today with a clean path to future releases. Primary site runs RAC for local HA, second site holds a synchronous standby for zero-data-loss failover, third site holds an asynchronous standby for regional disaster recovery.

Design principle

Tiered service classes

Not every database earns RAC. Classify the estate into tiers (RAC, single instance HA, standard) so licensing and hardware are spent where the business impact justifies them.

Design principle

Version strategy

Land on 19c as the stable base, design Grid and storage so the 23ai/26ai upgrade is a rolling, low-drama event inside the same platform lifecycle.

Design principle

Everything rehearsed

Switchover, failover, restore, and patch cycles are drilled on schedule and produce evidence artifacts, which doubles as NERC CIP recovery-plan compliance.

Recommendation logic

Lead with engineered systems, price the fallback

For a NERC CIP utility with strict HA/DR and a 2033 horizon, an engineered platform minimizes integration risk, patch complexity, and audit surface. Carrying a priced converged alternative in the response proves diligence and protects on cost.

Migration factory: pilot, then waves

The estimate becomes credible when the migration is presented as a repeatable production line with rollback gates, not a heroic weekend.

0
Discovery and wave designFull estate inventory, dependency mapping, downtime tolerance per app, method selection per database. Output: wave catalog and master runbook templates.
1
Pilot migrationOne representative tier-2 database end to end: build, migrate, validate, cut over, roll back once on purpose. The pilot hardens the runbook and calibrates the effort model with real timings.
2
Waves 1..N, grouped by dependency and criticalityStandard method per class: Data Guard switchover for like-for-like 19c moves, RMAN duplicate or transportable tablespace for restructures, Data Pump for small schemas, GoldenGate option for near-zero-downtime tier-1 cutovers.
3
Validation and acceptance per waveRow-count and object parity checks, performance baseline vs AWR reference, application smoke tests, signed acceptance gate before source decommission scheduling.
4
Rollback strategy, always armedSource kept in guaranteed-restore state through the acceptance window. Every runbook carries an explicit rollback trigger, owner, and time budget.
5
Knowledge transfer, training, hypercareRunbook walkthroughs, operations training, DR drill with client staff driving, then a defined hypercare window with severity-based response before steady-state support handover.

Key assumptions to state
  • Client provides network links between sites meeting stated latency budget
  • Existing Oracle licenses transferable to target platform, or delta priced separately
  • Application remediation (connection strings, drivers) owned by app teams with our runbook guidance
  • Facilities, power, and rack space ready per install schedule
Recommended exclusions and options
  • Application code changes and performance rewrites: excluded
  • Non-Oracle database migrations: excluded, quotable separately
  • Oracle 26ai upgrade program: priced option
  • GoldenGate implementation and support: priced option
  • Steady-state managed support post-hypercare through 2033: priced option, annual terms
